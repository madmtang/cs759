\begin{abstract}
DeepSeek Sparse Attention (DSA) reduces the main attention computation in long-context
LLM inference by selecting a fixed number of key-value entries per query token before
running attention. This report studies the kernel-level part of that computation: given
the selected token indices, compute the DSA attention output and log-sum-exp values for
the DeepSeek-V3.2-Exp-style MLA/MQA layout. Starting from a PyTorch reference
implementation, I developed a two-stage Triton solution that materializes logits once,
computes base-2 LSE in the first stage, and reuses the saved logits to form the output
without recomputing query-key dot products for every value tile. On the Euler cluster
GPU benchmark, the solution is correct on all 23 workloads and reaches a reported mean
speedup of $26.17\times$ with average latency reduced from about $9.13$ ms to
$0.368$ ms.
\end{abstract}

\section{Problem Statement / Introduction}
\label{submission}

The initial project proposal targeted optimized attention kernels for LLM inference,
with sparse attention as the main case study. The motivation was that attention becomes
one of the dominant costs in long-context inference: dense causal attention grows
quadratically with sequence length, and even optimized dense kernels must move and
combine a large amount of KV-cache data. The proposal therefore planned to study
DeepSeek Sparse Attention (DSA), implement relevant Triton kernels, and benchmark
them against a reference implementation.

The final project scope narrowed in a useful way. Instead of implementing the full DSA
pipeline, including the lightning indexer and top-$k$ selector, this work focuses on the
core sparse-attention kernel after the selected token indices are already available. That
is the part represented by \texttt{baselines/attention.py} and the task
\texttt{dsa\_sparse\_attention}, with 16 query heads, 512 compressed-cache dimensions,
64 positional-key dimensions, 2048 selected tokens, and 64-token pages. The upstream
indexer determines which tokens should be attended to; the kernel studied here must
efficiently gather those tokens from the paged cache, compute attention logits for all
query heads, produce a numerically stable softmax, and return both the BF16 output and
base-2 log-sum-exp (LSE).

\subsection{DSA Background}

DeepSeek-V3.2-Exp introduces DSA as a sparse-attention modification to
DeepSeek-V3.1-Terminus \cite{deepseekv32}. The DSA design has two conceptually
separate components. First, a lightweight indexer assigns an index score $I_{t,s}$
between a query token $t$ and each preceding token $s$. Second, a fine-grained selector
keeps only the key-value entries with top-$k$ index scores, and the main attention
computation runs over this selected set. In the sparse training stage described in the
DeepSeek-V3.2-Exp report, $k=2048$, matching the benchmark used in this project.

This architecture changes the main attention complexity from $O(L^2)$ to $O(Lk)$ for
sequence length $L$, while leaving a much smaller indexer computation upstream. The
paper also instantiates DSA under Multi-head Latent Attention (MLA) \cite{deepseekv2}
in an MQA-like mode \cite{shazeer2019}, where one latent key-value entry is shared
across multiple query heads. That detail is directly reflected in the benchmark layout:
each query token has 16 query heads, but the paged cache stores one 512-dimensional
compressed key/value vector and one 64-dimensional positional key vector per cached
token. This is also consistent with hardware-aware sparse-attention work that emphasizes
aligning sparse patterns with GPU execution constraints \cite{yuan2025nsa}.

\subsection{Kernel Interface}

For each query token $t$ and head $h$, the reference implementation computes over
the selected token set $S_t$ encoded by \texttt{sparse\_indices[t]}. A selected global
token id $s$ maps to page $p=\lfloor s/64\rfloor$ and page offset $r=s\bmod 64$.
The benchmark constants are:
\[
\begin{aligned}
H &= 16,\quad d_c = 512,\quad d_r = 64, \\
k &= 2048,\quad \text{page size} = 64.
\end{aligned}
\]
The logit for one selected token is
\[
\ell_{t,h,s} =
\alpha\left(
q^c_{t,h}\cdot c_{p,r} + q^r_{t,h}\cdot k^r_{p,r}
\right),
\]
where $\alpha$ is the softmax scale, $c_{p,r}$ is the compressed latent cache entry
used as both key content and value, and $k^r_{p,r}$ is the positional key. The output is
\[
o_{t,h}=\sum_{s\in S_t}
\frac{\exp(\ell_{t,h,s})}{\sum_{u\in S_t}\exp(\ell_{t,h,u})}c_{p(s),r(s)}.
\]
The LSE value returned by the benchmark is
\[
\mathrm{LSE}_{t,h}=\log_2\sum_{s\in S_t}\exp(\ell_{t,h,s}).
\]

\section{Solution Description}

The baseline in \texttt{baselines/attention.py} is a PyTorch implementation with a
token-level Python loop. For each token, it filters invalid \texttt{-1} entries, gathers
the selected rows from the flattened paged caches, computes the two logit terms with
matrix multiplications, runs \texttt{torch.logsumexp} and \texttt{torch.softmax}, and
then writes the BF16 output. From a performance perspective, this materializes
intermediate tensors such as the gathered keys, logits, and probabilities. The repeated
gather and softmax work across 16 heads and 2048 selected tokens is the main target for
GPU fusion.

The final solution in \texttt{results/solution.py} uses two Triton kernels. The first
kernel computes all logits and the LSE for one token-head pair. The second kernel
computes output tiles by reusing the saved logits and gathering only the value vectors.
This design deliberately stores a \texttt{[num\_tokens, 16, 2048]} FP32 logit tensor.
That is a different tradeoff from a recomputation-heavy design: it spends additional
global memory bandwidth to avoid recomputing the expensive 512-dimensional compressed
key dot product and 64-dimensional positional-key dot product eight times, once for
each 64-element output tile. For this benchmark, the saved-logit design is the better
choice because each query/head has a fixed top-$k$ of 2048 and the output dimension is
512.

\subsection{Stage 1: Full-Top-$k$ Logit and LSE Kernel}

The first Triton kernel launches a one-dimensional grid with one program per
$(t,h)$ pair:
\[
\texttt{grid1}=(\texttt{num\_tokens}\times 16).
\]
Inside each program, \texttt{BLOCK\_K=2048}, so the full sparse selection for that
token is processed in one vectorized program instance. This removes the need for an
online block-by-block softmax update over the selected-token dimension. Instead, the
kernel computes the complete logit vector, masks invalid selected-token entries to
$-\infty$, stores the FP32 logits, and then computes the LSE with one vector reduction:
\[
m=\max_s \ell_s,\qquad
\mathrm{LSE}=\log_2\sum_s \exp(\ell_s)
          = m\log_2(e)+\log_2\sum_s 2^{(\ell_s-m)\log_2(e)}.
\]
The explicit base-2 form matches the benchmark output contract and uses \texttt{exp2}
and \texttt{log2}, which are natural operations for the LSE representation used by the
evaluator.

The compressed-cache dot product is tiled with \texttt{BLOCK\_D=64}. Each program
therefore performs eight 64-wide reductions to cover the 512-dimensional compressed
key. The positional-key path uses \texttt{BLOCK\_PE=32}, requiring two 32-wide
reductions for the 64-dimensional RoPE/positional component. Both parts use the same
page/offset mapping:
\[
p=\lfloor s/64\rfloor,\qquad r=s\bmod 64,
\]
with the implementation using an equivalent bitwise offset computation because the page
size is fixed at 64. Invalid \texttt{-1} entries are mapped to a safe index for address
calculation but are masked out of all loads and reductions, so padding entries cannot
affect logits or LSE values.

\subsection{Stage 2: Logit Reuse for Output Tiles}

The second Triton kernel launches one program per token, head, and 64-dimensional output
tile:
\[
\texttt{grid2}=(\texttt{num\_tokens}\times 16\times 8).
\]
Each program accumulates one slice of the 512-dimensional output vector. It reads the
base-2 LSE from stage 1 and scans the selected-token dimension in \texttt{BLOCK\_K=128}
chunks. For each chunk, it loads the stored natural-logits, converts them into softmax
weights with
\[
w_s = 2^{\ell_s\log_2(e)-\mathrm{LSE}},
\]
gathers the corresponding value slice from the compressed cache, and accumulates
\[
o_{t,h,d:d+63} \mathrel{+}= \sum_s w_s c_{p(s),r(s),d:d+63}.
\]
Only the value path is repeated across the eight output tiles. The query-key and
query-positional dot products are not repeated in stage 2, which is the main reason for
materializing logits in stage 1.

\subsection{Kernel Specialization and Data Movement}

The implementation specializes to the fixed benchmark shape: 16 query heads,
512-dimensional compressed cache vectors, 64-dimensional positional vectors, 2048
selected tokens, and 64-token pages. These constants simplify program indexing and
avoid general-shape branches inside the Triton kernels. Stage 1 uses eight warps because
one program owns the full 2048-token logit vector and performs wide reductions. Stage 2
uses four warps because each program handles a smaller 128-token by 64-value tile.

The wrapper also normalizes input placement and types before launching Triton. Query and
cache tensors are made contiguous BF16 tensors on the target CUDA device, sparse indices
are converted to contiguous INT32, and the scalar softmax scale is passed as a CUDA FP32
scalar. This keeps the kernel body focused on the attention computation and avoids
unexpected stride, dtype, or CPU/GPU placement mismatches during evaluation.

\section{Overview of Results / Evaluation}

The current results were measured on the GPU node available through the University of
Wisconsin--Madison Euler cluster. The benchmark is the same DSA sparse-attention task
with 23 workloads. The result file reports that the Triton solution compiled, passed
numerical correctness, and was evaluated against the benchmark reference implementation
on the same Euler-cluster GPU.

The reported speedup is the benchmark's mean of per-workload speedup factors. This is
slightly different from dividing the average reference latency by the average solution
latency, but both views show the same conclusion: the final kernel substantially
reduces runtime for this task.

The average reference latency is $9.132$ ms, while the final Triton solution runs in
$0.368$ ms on average. Using these averaged latencies directly, the runtime is reduced
by about $96.0\%$. The benchmark's reported mean speedup is $26.17\times$, which is
larger than the ratio of the two averages because the evaluator averages per-workload
speedup factors. The correctness statistics remain within the evaluator tolerance: the
reported absolute error is $0.00320$ and the reported relative error is $0.04477$.

The performance result is consistent with the design choices in the solution. Stage 1
does the expensive query-cache and query-positional dot products exactly once per
token/head and writes the resulting logits. Stage 2 then uses those logits for all eight
output tiles, so the only repeated work is reading probabilities and value slices. This
is a good fit for the DSA benchmark because the selected-token dimension is large
($k=2048$) and the value dimension is also large (512). Recomputing logits for every
output tile would repeat the 576-dimensional dot-product path eight times, while the
stored-logit design replaces that repeated arithmetic with a sequential FP32 logit read.

The remaining bottleneck is likely the indirect gather pattern from the paged KV cache.
Sparse token ids can point to arbitrary pages and offsets, so the kernel cannot rely on
the same memory coalescing pattern as dense attention over contiguous sequence blocks.
The final implementation reduces overhead around this gather by specializing constants,
using fixed 64-wide value tiles, and avoiding unnecessary shape-general code in the
Triton kernels. The result illustrates the central kernel-level issue in DSA: after the
algorithm has reduced attention to a fixed top-$k$, performance depends on turning
irregular selected-token gathers into enough parallel work to keep the GPU occupied.

\section{Conclusions and Future Work}

This project started with a broad plan to study sparse attention, top-$k$ selection, and
Triton implementation techniques. The final work concentrated on the DSA core-attention
kernel, which was the right level of scope for making concrete GPU-performance progress.
The main result is a two-stage Triton implementation that stores logits once, reuses
them across output tiles, and reaches a reported $26.17\times$ mean speedup on the
Euler-cluster GPU benchmark.

The most useful optimization lesson was that the right memory/arithmetic tradeoff is
shape-dependent. The sparse token set is fixed at 2048 entries, and each entry requires
a 512-dimensional content dot product plus a 64-dimensional positional dot product. In
this setting, storing the logits is worthwhile because it prevents stage 2 from repeating
that dot-product work for every output tile. Specializing known constants such as the
head count, page size, and feature dimensions also removes overhead without changing the
algorithm.

Future work should profile the best kernel with hardware counters to separate memory
bandwidth, L2 behavior, register pressure, and occupancy effects. The current two-pass
design stores a full FP32 logit tensor, so a natural next experiment is to compare this
against hybrid designs that store smaller partial summaries or split the selected-token
dimension across programs with an explicit reduction. Another direction is to optimize
the upstream lightning indexer in \texttt{baselines/indexer.py} and study the full DSA
pipeline, including FP8 dequantization and top-$k$ selection.
Finally, this work directly uses ME759 topics: GPU thread/block decomposition, memory
hierarchy awareness, tiling, performance benchmarking, and parallel algorithm design for
modern ML inference.
